В качестве примера используется простое микро-веб-приложение на Flask, которое запускает HTTP-сервер и при обращении к нему выводит сообщение:

"Yesss, i did it!"

# Часть 1

## Dockerfile

## 1. Использование образа `latest`

В плохом Dockerfile в качестве базового образа используется: 

FROM python:latest

На первый взгляд это удобно, однако это вредно. Ведь это не указывает конкретную версию образа, поэтому при каждой новой сборке Docker может скачать 
другую версию образа. Что делает сборку нестабильной и непредсказуемой. Кроме того, образ `latest` обычно значительно больше по размеру, так как содержит 
большое количество дополнительных пакетов.

Нужно следующий образом, если использовать питон: 

FROM python:3.11-slim

## 2. Копирование всего проекта в контейнер

В `Dockerfile.bad` используется команда: 

COPY . /app

Такая команда копирует абсолютно все файлы проекта в контейнер.  
Это приводит к нескольким проблемам. Во 1-ых, могут копироваться файлы, которые не нужны для работы приложения. Во 2-ых, это нарушает механизм кеширования Docker.
Например, если изменится любой файл проекта (даже README), Docker будет считать слой изменённым и заново устанавливать все зависимости.
В исправленном варианте сначала копируется файл зависимостей, затем устанавливаются библиотеки, и только после этого копируется код приложения:

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .

---

## 3. Установка зависимостей без контроля кеша

В плохом зависимости устанавливаются напрямую: 

RUN pip install flask

Это считается плохим вариантом, так как зависимости проекта не фиксируются, невозможно воспроизвести точное окружение и кеш pip остаётся внутри образа, что увеличивает его размер.

В исправленной версии для этого используется файл `requirements.txt`, а также отключается кеш pip:

RUN pip install --no-cache-dir -r requirements.txt

---

## 4. Запуск контейнера от root

По умолчанию Docker запускает контейнер от пользователя `root`.  
Это удобно, но - небезопасно, так как контейнер получает слишком большие права.

Поэтому в хорошем варианте для этого специально создаётся отдельный пользователь:

RUN useradd -m appuser
USER appuser

---

## 5. Использование встроенного Flask-сервера

В приложении используется встроенный сервер 

Flask: app.run()

Стоит отметить, что этот сервер предназначен только для разработки. Он не рассчитан на обработку большого количества запросов и не является production-решением. 
В реальных проектах обычно используются WSGI-серверы.

---

## 6. Большое количество RUN-слоёв

Ещё одной распространённой проблемой является создание большого количества слоёв Docker.
Например:

RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y git

Каждая команда создаёт новый слой образа, что увеличивает его размер.

Лучше - объединение команд:

RUN apt-get update
&& apt-get install -y curl git
&& rm -rf /var/lib/apt/lists/*

Это уменьшает количество слоёв и делает образ компактнее.

## Плохие практики при работе с контейнерами

Следует не забывать, что контейнеры должны быть воспроизводимыми и легко управляемыми. Однако на практике иногда контейнеры могут быть использованы неправильно, что вполне легко может привести к проблемам при работе с проектом.

К одной из таких относится - ручное изменение контейнера после его запуска. Например, заход внутрь контейнера с помощью команды `docker exec -it container bash` и установка доп пакетов или изменения файлов. Такие изменения существуют только внутри данного контейнера и не сохраняются в образе. Поэтому если контейнер будет удалён и создан заново, все внесённые изменения - просто потеряются. По этой причине следует вносить все изменения непосредственно в Dockerfile, а затем пересобирать образ.

Ещё одна проблема - запуск контейнеров без указания имени. Если контейнер запускается обычной командой docker run, Docker автоматически присваивает ему случайное имя. Такие имена неудобны при работе с несколькими контейнерами, так как ими становится сложно управлять. Поэтому рекомендуется задавать имя контейнера вручную с помощью параметра `--name`, что упрощает работу с контейнерами и делает её более понятной, да и приятной.


# Часть 2

## Docker Compose

В этой части лабораторной рассматриваются плохие практики при написании `docker-compose` файлов.  
Docker Compose используется для запуска нескольких контейнеров одновременно и управления ими как одним проектом.

---

## 1. Использование образа `latest`

В плохом compose-файле используется образ базы данных:

image: postgres:latest

Использование тега `latest` плохо, так как он не фиксирует конкретную версию приложения.  
При следующей сборке Docker может скачать новую версию образа, что может привести проблемам и неработоспособности. О чем уже говорили в первой части

В исправленном варианте используется конкретная версия PostgreSQL:

image: postgres:15

---

## 2. Слабые учетные данные базы данных

В плохом compose-файле используются слишком простые учетные данные:

POSTGRES_USER: admin
POSTGRES_PASSWORD: admin

В качественном варианте используются суперзащищенные данные:

POSTGRES_USER: myuser
POSTGRES_PASSWORD: strongpassword

Это просто уменьшает риск несанкционированного доступа к данным.

---

## 3. Отсутствие volume для базы данных

В плохом compose-файле контейнер базы данных запускается без volume.

Это означает, что все данные базы будут храниться только внутри контейнера.  
Если контейнер будет удалён или пересоздан, все данные базы данных будут потеряны.

Поэтому в исправленном варианте используется volume:

volumes:
postgres-data:/var/lib/postgresql/data

Это позволяет сохранять данные базы даже после перезапуска или удаления контейнера.

---

## 4. Отсутствие restart policy

В плохом compose-файле не задана политика перезапуска контейнеров.  
Если контейнер завершит работу из-за ошибки или сбоя, он не будет автоматически запущен снова.

В исправленном варианте используется:

restart: always

Теперь контейнер автоматически перезапускается при сбоях.

---

## 5. Отсутствие сетевой изоляции сервисов

По умолчанию Docker Compose помещает все сервисы в одну сеть, что означает, что контейнеры могут свободно взаимодействовать друг с другом.
В некоторых случаях это нежелательно, так как усложняет управление инфраструктурой.

Поэтому следует разделять по разным сетям:

networks:
web-network
db-network

Контейнер `web` подключен к сети `web-network`, а контейнер `db` - `db-network`.

Таким образом контейнеры запускаются в рамках одного compose-проекта, но не имеют прямого сетевого доступа друг к другу.
Это позволяет изолировать сервисы и повысить безопасность системы.
