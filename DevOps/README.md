# Часть 1

## Dockerfile

## 1. Использование образа `latest`

В плохом Dockerfile в качестве базового образа используется: 

FROM python:latest

На первый взгляд это удобно, однако это обман. Если не указывается конкретная версия образа, то при каждой новой сборке Docker-а происходит скачивание 
последний версии образа. Что делает сборку нестабильной и непредсказуемой. Кроме того, образ `latest` значительно больше по размеру (как видно на фото), 
так как он содержит большое количество дополнительных пакетов.

Писать нужно следующий образом (если на python): 

FROM python:3.11-slim

## 2. Копирование всего проекта в контейнер

В `Dockerfile.bad` используется команда: 

COPY . /app

Такая команда копирует абсолютно все файлы проекта в контейнер.  
Это приводит к нескольким проблемам. Во 1-ых, могут копироваться файлы, которые не нужны для работы приложения. Во 2-ых, это нарушает механизм кеширования Docker.
Например, если изменится любой файл проекта (README в том числе), Docker будет считать слой измененным и заново устанавливать все зависимости.
В правильном варианте сначала копируется файл зависимостей, затем устанавливаются библиотеки, и только после этого копируется код приложения:

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .

---

## 3. Установка зависимостей без контроля кеша

В плохом случае зависимости устанавливаются напрямую: 

RUN pip install flask

Это так, ввиду того, что зависимости проекта не фиксируются, невозможно воспроизвести точное окружение и кеш pip остаётся внутри образа, что еще и увеличивает его размер.

В исправленной варианте для этого используется файл `requirements.txt`, а также отключается кеш pip:

RUN pip install --no-cache-dir -r requirements.txt

---

## 4. Запуск контейнера от root

По умолчанию Docker запускает контейнер от пользователя `root`.  
Это удобно, но - небезопасно, так как контейнер получает слишком большие права.

Поэтому в хорошем варианте для этого специально создаётся отдельный пользователь:

RUN useradd -m appuser
USER appuser

---

## 5. Использование встроенного Flask-сервера

В нашем миниприложении используется встроенный сервер: 

Flask: app.run()

Можно отметить, что этот сервер предназначен только для разработки, так как не рассчитан на обработку большого количества запросов. 
В реальных проектах обычно используются WSGI-серверы.

---

## 6. Большое количество RUN-слоёв

Ещё одной распространённой проблемой является создание большого количества слоёв Docker.

Например:

RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y git

Каждая команда создаёт новый слой образа, что увеличивает его размер.

Лучше - объединение команд:

RUN apt-get update
&& apt-get install -y curl git
&& rm -rf /var/lib/apt/lists/*

Это уменьшает количество слоёв и делает все компактнее.

## Плохие практики при работе с контейнерами

Следует не забывать, что контейнеры должны быть воспроизводимыми и легко управляемыми. Однако иногда контейнеры могут быть использоваться неправильно, что легко приводить к проблемам при работе с проектом.

К одной из таких относится - ручное изменение контейнера после его запуска. Например, заход внутрь контейнера с помощью команды `docker exec -it container bash` и установка доп пакетов или изменения файлов. Такие изменения существуют только внутри данного контейнера и не сохраняются в образе. Поэтому если контейнер будет удалён и создан заново, все внесённые изменения - просто потеряются. По этой причине следует вносить все изменения непосредственно в Dockerfile, а затем пересобирать образ.

Еще одна проблема - запуск контейнеров без указания имени. Если контейнер запускается обычной командой docker run, Docker самостоятельно присваивает ему случайное имя. Такие имена неудобны при работе с несколькими контейнерами, так как ими становится сложно управлять. Поэтому следует задавать имя контейнера вручную с помощью параметра `--name`. Это упрощает работу и делает её более понятной, да и приятной, что скрывать.


# Часть 2

## Docker Compose

В этой части лабораторной рассматриваются плохие практики при написании `docker-compose` файлов.  
Docker Compose используется для запуска нескольких контейнеров одновременно и управления ими как одним проектом.

---

## 1. Использование образа `latest`

В плохом compose-файле используется образ базы данных:

image: postgres:latest

Использование тега `latest` плохо, так как он не фиксирует конкретную версию приложения.  
При следующей сборке Docker вполне может скачать новую версию образа, что может привести проблемам и неработоспособности. О чем уже говорили в первой части

В хорошем варианте используется конкретная версия PostgreSQL:

image: postgres:15

---

## 2. Учетные данные базы данных

В плохом compose-файле используются базовые учетные данные, которые не защищают от несанкц доступа к даннным:

POSTGRES_USER: admin
POSTGRES_PASSWORD: admin

Поэтому следует использовать супернепредсказуемые данные):

POSTGRES_USER: myuser
POSTGRES_PASSWORD: strongpassword


---

## 3. Отсутствие volume для базы данных

В плохом compose-файле контейнер базы данных запускается без volume.

Значит, что все данные базы будут храниться только внутри контейнера.  
Если контейнер будет удалён или пересоздан, все данные базы данных будут потеряны.

Поэтому в исправленном варианте используется volume, что позволяет сохранять данные базы 
даже после перезапуска или удаления контейнера:

volumes:
postgres-data:/var/lib/postgresql/data

---

## 4. Отсутствие restart policy

В плохом файле не задана политика перезапуска контейнеров.  
И если контейнер завершит работу из-за ошибки или сбоя, он не будет автоматически запущен снова.

Поэтому следует указывать автоперезапуск следующим образом:

restart: always

---

## 5. Отсутствие сетевой изоляции сервисов

По умолчанию Docker Compose помещает все сервисы в одну сеть, то есть - контейнеры могут свободно взаимодействовать друг с другом.
Иногда это нежелательно, так как усложняет управление.

Поэтому лучше разделять по разным сетям:

networks:
web-network
db-network

Контейнер `web` подключен к сети `web-network`, а контейнер `db` - `db-network`.

Так контейнеры запускаются в рамках одного compose-проекта, но не имеют прямого сетевого доступа друг к другу.
