## Часть 2. Helm Chart

### 1. Создание Helm-чарта

Во второй части работы те же ресурсы Kubernetes были упакованы в Helm Chart.  
Для этого я был создан новый чарт с помощью команды: helm create my-app
После этого была получена стандартная структура чарта: папка templates, файл values.yaml и файл Chart.yaml.

<img width="1334" height="341" alt="image" src="https://github.com/user-attachments/assets/b31204da-43a5-41d5-848a-f39f267e8880" />
Скрин 5 - установка Helm и создание Helm-чарта

### 2. Установка чарта

Далее приложение задеплоил в кластер с использованием Helm: helm install my-release ./my-app

Helm автоматически сгенерировал Kubernetes-манифесты на основе шаблонов и применил их в кластер. После установки был проверен список релизов, чтобы убедиться, что всё задеплоилось корректно.

### 3. Обновление приложения (upgrade)

После этого я изменил чуть-чуть текст, чтобы проверить работу обновления через Helm.

Затем выполнил команду: helm upgrade my-release ./my-app

Helm обновил релиз, и была создана новая версия (revision), что можно увидеть в выводе команды.

В ходе выполнения работы стало понятно, что Helm значительно упрощает работу с Kubernetes по сравнению с обычными YAML-манифестами.

#### Во-первых, 
Helm позволяет удобно работать с параметрами. В стандартных манифестах все значения жёстко прописаны, и при любом изменении приходится редактировать сами YAML-файлы. А в Helm же основные настройки выносятся в файл values.yaml, благодаря чему можно быстро менять параметры приложения, не трогая сами шаблоны.

#### Во-вторых, 
упрощается процесс обновления приложения. Вместо повторного применения манифестов используется команда helm upgrade, которая автоматически применяет изменения и создаёт новую версию релиза

#### Наконец, 
Helm позволяет управлять версиями приложения. Каждое обновление сохраняется как новая ревизия, благодаря чему при необходимости можно откатиться к предыдущему состоянию. В обычных манифестах такой возможности нет, поэтому всё приходится отслеживать вручную.
