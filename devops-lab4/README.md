# Часть 1. CI/CD пайплайны

Я реализовал два варианта пайплайна:
- плохой (bad-ci.yml)
- хороший (good-ci.yml)

Идея была в том, чтобы сначала показать типичные ошибки, а затем исправить их.

## Плохие практики (bad-ci.yml)

### 1. Деплой при любых условиях

Проблема:  
Деплой выполняется даже если предыдущие этапы упали.

Почему это плохо:  
В продакшен может попасть сломанный код.

Как исправлено:  
В хорошем пайплайне деплой зависит от успешного прохождения всех этапов.

### 2. Хранение секретов в коде или env

Проблема:  
Секреты передаются напрямую через переменные окружения.

Почему это плохо:  
Они могут попасть в лог или быть скомпрометированы.

Как исправлено:  
Используются GitHub Secrets (`${{ secrets.* }}`).

### 3. Использование `ubuntu-latest`

Проблема:  
Окружение не фиксировано.

Почему это плохо:  
После обновления runner пайплайн может внезапно сломаться.

Как исправлено:  
Используется конкретная версия — `ubuntu-22.04`.

### 4. Отсутствие зависимостей между этапами

Проблема:  
Этапы могут выполняться параллельно.

Почему это плохо:  
Деплой может начаться до завершения тестов.

Как исправлено:  
Добавлены зависимости через `needs`.

### 5. Повторная установка зависимостей

Проблема:  
Зависимости устанавливаются каждый раз заново.

Почему это плохо:  
Замедляет выполнение пайплайна.

Как исправлено:  
Добавлено кэширование (в хорошем пайплайне).


## Хороший пайплайн (good-ci.yml)

В исправленной версии:

### ✔ Чёткая последовательность этапов: lint → test → security → build → deploy

Каждый этап выполняется только после предыдущего.

### ✔ Фиксированное окружение

Используется: runs-on: ubuntu-22.04
Это делает пайплайн стабильным и воспроизводимым.

### ✔ Безопасная работа с секретами

Секреты не хранятся в коде и не передаются напрямую.  
Используется GitHub Secrets: ${{ secrets.DEPLOY_TOKEN }}

### ✔ Отказ от небезопасных конструкций
- нет `if: always()`
- нет игнорирования ошибок

### ✔ Более контролируемый запуск

Пайплайн запускается только при изменениях в коде, а не при каждом коммите.

# Часть 2. Работа с секретами

Во второй части я сделал простую демонстрацию работы с секретами через Docker.

---

## Идея

Есть два сервиса:
- Vault (хранилище секретов)
- Python-приложение (клиент)

Приложение получает секрет и использует его, но не выводит в открытом виде.

## Запуск

```bash
cd devops-lab4/vault-demo
docker compose up
Результат работы
```

В логах видно:

Reading secret safely...
Secret masked: ********

Это означает, что секрет используется, но не раскрывается.

Почему это важно

Если секреты хранить:

в коде
или в переменных CI

то они могут:

попасть в лог
утечь
быть украдены
Что даёт Vault (в реальности)

Хотя в данной работе используется упрощённый сценарий, сам подход повторяет реальные системы.

Vault позволяет:

хранить секреты централизованно
выдавать временные токены
контролировать доступ
вести аудит
